# Modularization Review — trade-tms-go

Scope: analysis-only architectural review of `github.com/byjackchen/trade-tms-go` (~40 internal packages, five operating modes on one deterministic engine). Findings below survived adversarial verification (real defects/enhancements, not merely intentional design). Each cites `file:line`.

---

## 1. Executive Summary

**Overall health: strong.** The intended inner→outer layering is largely respected, the strategy/portfolio/accounting/engine core is genuinely shared, and the five-mode "swap only Clock + DataFeed + Executor" thesis holds in practice. The `domain` package depends on nothing, preserving the precondition for deterministic golden tests. Most of what verification surfaced is **documentation drift** and **incidental duplication**, not structural rot — there are **no import cycles**, **no illegal inner-imports-outer edges**, and the one L3→L3 edge (`exec/moomoo → engine`) is the idiomatic "adapter satisfies the consumer's port" pattern, not a defect.

**Top themes of weakness:**

1. **doc.go-vs-reality drift (highest-frequency theme).** `exec/doc.go` advertises a non-existent `ExecutionClient` interface and the false property "the engine cannot tell them apart"; `hyperopt/doc.go` calls nsga2/study "not yet built" when they are the largest, fully-wired part of the subtree; `adapters/doc.go` states a config-credential import rule the only real adapter ignores. Most packages lack the promised "May import" contract entirely, and 12 packages (including the 2186-LOC `runner` orchestrator) have no doc.go at all.
2. **Batch/streaming dispatch duplication across the engine seam.** `engine.Engine` and `livengine.Session` hand-copy context injection, warmup priming, and per-bar dispatch, held together by `// identical to engine.X` comments with **no cross-path equivalence test**. This is exactly the structure that produced the prior P4 warmup-divergence bug.
3. **Strategy-layer asymmetries.** `strategy/pairs` alone imports `params` (for dead, duplicated `FromParams`), pulling the full pgx driver into the pure-compute closure; SEPA/ORB leak local intent types up into the L4 `publish` layer while Pairs/Sector emit domain types cleanly.
4. **Consistency/hygiene.** A dead generated pb package, an underscore package name forcing aliases everywhere, three competing constructor/config conventions, and a copy-pasted Postgres test harness with divergent gating.

## 4. Recommended Sequencing

**Do first (high impact / low risk):**

1. **F1 + F5 + F7 + F4 — doc.go reconciliation pass.** Pure documentation, zero runtime risk, highest navigability payoff. `exec/doc.go`'s fictional `ExecutionClient` (F1) is the single most misleading artifact in the codebase — it sits at the center of the five-mode thesis. Bundle with the stale `hyperopt`/`adapters`/`adapters/moomoo` docs (F5, F6's note, F7) and the missing-doc.go sweep (F4). One focused PR; consider adding a depguard config so layering can't silently drift after.
2. **F2 — delete dead `pairs.FromParams` + split pgx out of `params`.** Deleting `build.go`'s `FromParams` is a one-line-impact change that removes the only pgx leak into the pure-compute closure and restores strategy symmetry. Mechanical, no behavior change (the symbol is dead). The pgx-reader split is slightly larger but the `PayloadReader` seam already exists.
3. **F3 (test half first) — add the batch-vs-streaming equivalence test.** Before any refactor, add the cross-path test that runs identical assembly output through `engine.New` and `livengine.Session.Replay`. This is the highest-value, lowest-risk half: it would have caught the prior P4 warmup bug and makes the "100% shared engine" thesis structurally guarded instead of comment-asserted. Defer the full dispatch-core extraction until the test exists.
4. **F8 / F9 — cheap consistency wins.** Renaming `sector_rotation → sectorrotation` (F8) and unifying integration-test gating (F9) are low-effort and remove real friction (alias noise; broken hermetic-`go test` promise).

**Defer / treat as contentious:**

- **E3 (SEPA/ORB intent leak)** — tempting to "fix" by making pure packages emit domain types, but that **violates the documented zero-domain layering invariant**. The only safe version is relocating the normalization into the adapters; do it deliberately, not as a drive-by.
- **E1, E4 (gate/close-path extraction)** — sound DRY wins, but they touch correctness-sensitive submission and sizing logic. Sequence them *after* F3's equivalence test exists so any behavioral drift is caught.
- **E6 (two-engines clarification)** — do NOT attempt to merge `engine.Engine` and `livengine.Session` into one hot-swappable engine; the split is intentional (batch fill-timing vs live order lifecycle). This is a doc clarification only.
- **E5, E8 (api store extraction, testutil)** — pure cleanup, no urgency; fold into adjacent work on those packages.
