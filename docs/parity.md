# Parity ledger — production-grade accuracy proof

`trade-tms-go` is a behavior-preserving port of the Python reference
`trade-multi-strategies`. "Production-grade accuracy" means every layer is
proven equal to the reference, not merely "close." This document is the
consolidated proof ledger across phases.

**Python is the offline ORACLE only.** Every parity check captures golden output
from the reference CPython venv (or shells to it behind a build tag) and asserts
the Go output matches. None of these harnesses are part of the shipped image or
the default `go test ./...` run — they are committed golden fixtures plus
build-tag-gated gates (`parity`, `parity_folds`). Production has **zero Python
runtime dependency**.

## 1. Fixed-point money parity

USD amounts are `int64` fixed-point at 1e-4 scale (`internal/domain` `Money`),
the storage form throughout the DB. `fixed_test.go` proves exact round-tripping
and the Sharadar price-cap handling (the `_NAUTILUS_PRICE_MAX` value scales to
~1.7e17, comfortably inside int64; unrepresentable cells store NULL and are
counted, per `data-sharadar.md`). The float64 round-trip gate is scoped to the
parquet layer.

## 2. Indicator parity (≤ 1e-9)

Every technical indicator matches the numpy/pandas reference within `tol = 1e-9`.
The golden fixtures (`testdata/golden.json`) are regenerated by
`tmp/indicators_parity/gen.py` against the reference venv. Both the batch and the
incremental (streaming) code paths are checked, so the live per-bar update equals
the backtest batch computation. JSON `null` maps to NaN (the Python sentinel),
preserving NaN positions exactly.

## 3. Data field parity

The Sharadar sync + parquet import mirror the reference cache field-for-field:
the ticker universe filter (survivorship policy), the SEP/SFP bar layout, and
the SF1 fundamentals are validated against captured reference output
(`internal/data/sharadar`, `internal/data/universe/golden_test.go`). NaN-bearing
source cells map to NULL in the store; NaN-bearing tickers are dropped by
consumers, never silently cleaned.

## 4. Strategy signal parity — 0 mismatch

The four ported strategies (SEPA, sector_rotation, ORB/sector-orb, Pairs) are
signal-parity-proven against the reference: over the shared bar inputs, the Go
strategies emit the SAME signals as the Python strategies with **zero
mismatches**. This is anchored by the strategy-assembly gate
(`internal/engine/strategyassembly/gate_test.go`) and the centralized-params
baseline tests (`internal/params/baseline_test.go`,
`internal/params/resolve_test.go`), which prove the params resolve identically.

## 5. Fill / engine parity vs Nautilus (`parity` gate)

The permanent golden-parity integration gate (`internal/parity/parity_gate_test.go`,
build tag `parity`) runs the reference Nautilus harness and the Go engine over
the **same** `bars.json` + order script and diffs the artifacts with
`tmp/parity/compare_engine.py`. The comparator exits non-zero on any divergence
beyond tolerance:

- **Prices: exact** after fixed-point conversion.
- **PnL / equity: within a cent.**
- **Fill counts and ordering: exact.**

It emits `fills.json` (one entry per filled leg: ticker, side, qty, px, ts — the
exact per-fill price/qty/timing) and `equity.json` (per-bar total account
equity, comparable to the Nautilus `EquityCurveSamplerActor`). The depth-walk
fill model has its own parity test (`internal/engine/parity_depth_test.go`,
`internal/exec/fillmodel_depthwalk_test.go`). Run via `make parity`.

This gate also proves the **five-modes-one-engine** thesis: because backtest and
live share `internal/core`, a streaming run over a `VirtualClock` reproduces the
backtest bit-for-bit (`internal/livengine/consistency_test.go`).

## 6. Metrics parity (≤ 1e-12 relative)

Sharpe, Calmar, MaxDrawdown, volatility, and total return are computed with
Neumaier-compensated summation and agree with the reference to ≤ 1e-12 relative
on a 252-point randomized walk — the case where naive Go summation can drift in
the last ulp (`internal/metrics/parity252_test.go`). The walk-forward
return-stitching path has its own parity test (`stitch_test.go`). Reference
values were captured from `research.metrics` over the same seeded walk.

## 7. Hyperopt objective parity (`parity_folds` gate)

The NSGA-II walk-forward objective is proven against the reference
`research.workers` pipeline. `internal/hyperopt/study/parity_folds_test.go`
(build tag `parity_folds`) drives the Go objective evaluator over multiple real
adjacent folds with return-stitching and compares per-fold and stitched
Sharpe/Calmar/MaxDD against `tmp/parity_folds_harness.py`. This proves the
optimizer searches the same objective surface as the reference.

## 8. moomoo protocol byte-parity

The protocol conformance gate (`internal/adapters/moomoo/conformance_test.go`,
`conformance_trd_test.go`) asserts the Go client's **encoded request frames are
byte-for-byte identical** to the frames the vendored Python moomoo SDK produces
for the same request — same field values, same serial number, same header layout
+ SHA-1 + `struct.pack`. The reference bytes come from the SDK's own
`pack_pb_req` (`tmp/conformance/dump_frames.py`). The Go decoder is proven to
parse SDK-encoded reply and push frames (`TestDecodeSDK{InitConnectReply,
HistoryReply,UpdateKLPush}`), covering both `Qot_*` (quote) and `Trd_*` (trade)
paths. Client↔mock round-trip and reconnect/re-subscribe are covered in
`internal/adapters/moomoo/mock`. These run with NO OpenD — protocol fidelity is
proven offline.

## Reproducing the gates

```bash
make parity-tests                             # all three oracle gates at once

make parity                                   # full Nautilus + Go fill/equity gate
go test -tags parity ./internal/parity/       # the gate test itself

make parity-backtest                          # integrated-backtest gate (compose up)
make parity-folds                             # hyperopt objective gate (compose up)

# everything else (hermetic, golden fixtures, no venv needed):
go test ./...
```

Regenerating golden fixtures (from the read-only `trade-multi-strategies` venv —
the ORACLE; never modify it):

```bash
.venv/bin/python tmp/indicators_parity/gen.py   > internal/indicators/testdata/golden.json
.venv/bin/python tmp/conformance/dump_frames.py > internal/adapters/moomoo/testdata/conformance_frames.json
.venv/bin/python tmp/conformance/dump_replies.py> internal/adapters/moomoo/testdata/conformance_replies.json
```

The default `go test ./...` is hermetic: it consumes the committed golden
fixtures and never shells to Python. Only the `parity`, `parity_folds`, and
`parity_backtest` tags invoke the oracle, and none is part of the shipped image.

The three Python-touching gates are grouped under one Make target:

```bash
make parity-tests   # runs every build-tag-gated, Python-touching gate
                    # (= parity + parity-backtest + parity_folds);
                    # requires the read-only trade-multi-strategies venv.
```

## Zero Python runtime dependency — production audit

**Locked decision: production must have ZERO Python runtime dependency.** This
section is the standing audit that proves it; re-run the greps below after any
change that touches `cmd/`, `internal/`, the `Dockerfile`, or the test gates.

### What was audited

1. **Shipped binary / image.** The `Dockerfile` is a two-stage pure-Go build:
   `golang:1.26` builder → `CGO_ENABLED=0 go build` → `gcr.io/distroless/static-debian12:nonroot`
   final stage that contains ONLY the `tms` binary (migrations are embedded in
   the binary; nothing else is copied in). Distroless static has no shell, no
   package manager, and no Python interpreter. `.dockerignore` keeps `tmp/`,
   `.env*`, `docs/`, `bin/`, and `.git` out of the build context, so no parity
   harness or scratch Python ever enters even the builder layer.

2. **Production Go source.** Grepping `cmd/` + `internal/` (excluding `_test.go`)
   for `python | uv | .venv | os/exec | trade-multi-strategies` finds **only
   descriptive comments** ("matches the Python reference", "mirrors CPython
   statistics.mean", etc.). There is **no `os/exec` import in any non-test
   production file** and no runtime shell-out to Python, `uv`, or a `.venv`. The
   Go code reproduces the reference's numeric semantics natively (e.g.
   `internal/metrics` reimplements CPython exact-fraction summation in
   `math/big`); it never calls Python to do so.

3. **Default test run.** `go test ./...` (no tags) was executed with a PATH that
   contains **no `python`, `python3`, or `uv`** — all packages pass. No default
   test shells to the oracle: the parity assertions consume committed golden
   fixtures / embedded constants captured offline.

4. **Python-touching tests are all guarded.** The only tests that actually
   invoke the venv at runtime are behind build tags and are never compiled by the
   default run or the image:

   | File | Build tag | How it touches Python |
   |---|---|---|
   | `internal/parity/parity_gate_test.go` | `parity` | shells to `make parity` (Nautilus harness + comparator) |
   | `internal/hyperopt/study/parity_folds_test.go` | `parity_folds` | drives Go evaluator vs `tmp/parity_folds_harness.py` output |
   | `internal/hyperopt/study/parity_backtest_test.go` | `parity_backtest` | integrated-backtest vs embedded fresh-Python expectations |

   Other `*_test.go` files whose names/comments mention Python (e.g.
   `internal/params/baseline_test.go`, the moomoo `conformance_*_test.go`,
   `internal/strategy/*/golden_test.go`, `internal/metrics/parity252_test.go`)
   are **hermetic**: they embed the captured golden values as constants or read
   committed `testdata/` fixtures and never exec anything. Their Python
   references are regeneration instructions in comments, not runtime calls.

### Reproduce the audit

```bash
# (1) No runtime Python shell-out in production source:
grep -rnE 'python|/\.venv|"uv"|os/exec' cmd/ internal/ --include='*.go' \
  | grep -v '_test.go'          # → only comments, no os/exec import

# (2) Every Python-touching test is build-tag gated:
grep -rl 'exec\.Command\|\.venv\|make parity' --include='*_test.go' cmd/ internal/ \
  | xargs grep -L '//go:build'  # → empty (all such files carry a tag)

# (3) Default suite needs no interpreter — run with a Python-free PATH:
PATH=/usr/bin:/bin go test ./...   # (ensure no python on this PATH) → all ok

# (4) The image carries no Python:
docker build -t tms:audit .
docker run --rm --entrypoint /usr/local/bin/tms tms:audit version  # works
# distroless-static has no shell, so there is nothing to exec a python from.
```

If any future committed test shells to the oracle without a build tag, move it
behind one (`//go:build parity`) and add it to the `parity-tests` Make target;
do not let it leak into the default `go test ./...` or the shipped image.
